# Module 7 - Replatform To Azure Container Apps

## Target Choice

The selected managed container target is Azure Container Apps. It gives this monolith a managed container runtime with built-in HTTPS ingress, revision traffic, managed identity, secrets and replica scaling without the operational overhead of AKS.

## Deployment Shape

| Area | Configuration |
|---|---|
| Managed target | Azure Container Apps |
| Image | `petclinicacrc6mhqfua.azurecr.io/spring-petclinic:300c8f7` |
| Ingress | External HTTPS ingress to container port `8081` |
| Revision traffic | Single revision mode, 100% to latest revision |
| Scaling | Minimum 1 replica, maximum 2 replicas |

## Why Container Apps

| Criterion | Decision |
|---|---|
| Migration effort | Lower than AKS because no Kubernetes cluster operations are required |
| Runtime fit | Runs the existing containerized monolith with no application code changes |
| Ingress | Built-in HTTPS ingress meets the Module 7 endpoint requirement |
| Operations | Managed environment, revisions and scaling reduce platform burden versus VM and AKS |
| Future path | Can later attach managed database, VNet, Dapr, Key Vault and private ingress patterns |

## Operational Trade-Off Comparison

Detailed comparison is captured in [`inventory/07.01-replatform-target-comparison.csv`](../inventory/07.01-replatform-target-comparison.csv). The summary below is included directly in this document so the reviewer can see the operational decision without opening the CSV.

| Target | Operations Ownership | Release And Rollback | Network And Security | Cost / Scale Pattern | Recommendation |
|---|---|---|---|---|---|
| Azure VM | Highest ownership: OS patching, runtime, NGINX, backup, monitoring and hardening remain with the app/platform team. | Rollback usually requires redeploying artifacts, restoring snapshots/backups or switching traffic back to a previous VM. | Strong VNet and NSG control, but TLS, egress controls and Key Vault integration must be assembled and operated. | Always-on VM, disk, backup and monitoring cost; scaling requires VM resize, VMSS or manual automation. | Good Module 4 rehost baseline, but not the best Module 7 target because the app is already containerized. |
| Azure App Service for Containers | Low ownership: Azure manages the web host, front end and TLS endpoint; team manages image, app settings and scale rules. | Deployment slots give simple blue-green style swaps and fast rollback. | Managed identity and Key Vault references are mature; VNet integration is available, with less granular network control than AKS. | Predictable App Service Plan cost; straightforward instance scaling for steady web traffic. | Viable fallback when classic PaaS web operations and slots are preferred over Container Apps revisions. |
| Azure Container Apps | Balanced ownership: Azure manages the container environment, ingress, revisions and replica orchestration; team manages image, config and target network choices. | Revision traffic supports controlled rollout, traffic split and rollback to a prior healthy revision. | Supports managed identity, secrets, internal/external ingress and VNet-integrated managed environments; can be paired with Front Door, firewall, NAT Gateway and private endpoints. | KEDA-based min/max replicas and consumption/workload-profile pricing; can be efficient for variable traffic while keeping minimum replicas for availability. | Recommended for PetClinic because it meets managed container, ingress, secrets, revision traffic and scaling requirements without AKS cluster operations. |
| AKS | Highest platform-engineering ownership: node pools, upgrades, ingress controller, policies, add-ons and Kubernetes governance need an operating model. | Native rolling updates plus advanced progressive delivery tools are possible, but require mature cluster tooling. | Most flexible networking and policy model, including CNI choices, NetworkPolicy, private clusters, service mesh and workload identity. | Baseline cluster/node cost; efficient when many services share a governed Kubernetes platform. | Too heavy for this single monolith unless the enterprise already standardizes on AKS and has a platform team. |

The assessment recommendation remains Azure Container Apps. It gives enough modernization value over the VM path while avoiding the platform-engineering overhead of AKS. App Service for Containers is a credible alternative, but Container Apps better demonstrates revision traffic, container-native scaling and managed container ingress for this module.
