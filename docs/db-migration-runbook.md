# Database Migration Runbook

## Scope

This runbook covers Module 8 database migration for Spring PetClinic from the on-premises PostgreSQL profile database to Azure Database for PostgreSQL Flexible Server.

| Item | Value |
|---|---|
| Source database | `postgres-db-vm:5432/petclinic` |
| Target service | Azure Database for PostgreSQL Flexible Server |
| Selected mode | Offline `pg_dump` / `pg_restore` |
| Downtime tolerance | 30 minutes for the assessment workload |
| App config keys | `SPRING_PROFILES_ACTIVE=postgres`, `POSTGRES_URL`, `POSTGRES_USER`, `POSTGRES_PASS` |
| Cutover owner | Application / migration engineer |

## Source Dependency Summary

| Dependency | Finding |
|---|---|
| Engine | PostgreSQL selected; H2 is local/dev only; MySQL is optional profile support |
| Version | PostgreSQL 16 in local simulation; production source must be confirmed with `SELECT version();` |
| Size | Assessment seed data is less than 1 MB; production must be measured with `SELECT pg_size_pretty(pg_database_size('petclinic'));` |
| Schema count | 1 application schema |
| Object count | 33 logical objects: 7 tables, 7 indexes, 6 identity sequences and 13 constraints |
| Read/write ratio | Estimated 80/20 read/write from owner/vet browsing and owner/pet/visit update flows |
| Peak TPS | Local assessment baseline below 10 TPS; production value must be confirmed from APM or database metrics |
| Criticality | High for the application because owner, pet, vet and visit data are core state |

## Offline Versus Online Decision

| Mode | Fit | Decision |
|---|---|---|
| Offline dump/restore | Best for small data set and 30-minute downtime window | Selected |
| Azure DMS offline | Also valid but more setup than needed for this small PostgreSQL path | Not selected |
| Online DMS/logical replication | Best for near-zero downtime and larger production systems | Documented fallback |

Offline migration is selected because the PetClinic schema and seed data are small, the application can tolerate a short maintenance window in this assessment, and PostgreSQL-native tooling keeps the migration simple and auditable.

## Prerequisites

1. Azure Database for PostgreSQL Flexible Server exists in the target landing zone.
2. Target database `petclinic` exists.
3. Target network access is configured through private endpoint or approved firewall/VNet rules.
4. TLS is enforced on the target server.
5. Application secrets exist in Key Vault or platform secret store:
   - `petclinic-postgres-jdbc-url`
   - `petclinic-postgres-username`
   - `petclinic-postgres-password`
6. Migration workstation or runner has `pg_dump`, `pg_restore`, `psql` and network access to source and target.
7. Source writes can be stopped during the maintenance window.
8. Latest backup/PITR point is confirmed before cutover.

## Schema Conversion

The selected path is PostgreSQL to PostgreSQL. No SSMA cross-engine conversion is required.

Run the conversion assessment:

```powershell
python .\scripts\schema_convert\convert_schema.py
```

Expected output:

| Output | Purpose |
|---|---|
| `scripts/schema_convert/conversion_report.md` | Human-readable conversion report |
| `scripts/schema_convert/conversion_report.json` | Structured conversion findings |

Manual remediation notes:

| Source Pattern | Remediation |
|---|---|
| H2 `VARCHAR_IGNORECASE` | Use PostgreSQL `TEXT`; add `citext` or lower-case functional indexes only if case-insensitive semantics are required |
| MySQL `AUTO_INCREMENT` | PostgreSQL target uses `GENERATED BY DEFAULT AS IDENTITY` |
| MySQL `UNSIGNED` | Remove unsigned constraint or enforce with check constraints if required |
| MySQL `engine=InnoDB` | Not applicable in PostgreSQL |

## Offline Migration Steps

Set variables in the migration shell:

```powershell
$sourceHost = "postgres-db-vm"
$sourcePort = "5432"
$sourceDb = "petclinic"
$sourceUser = "petclinic"
$targetHost = "<postgres-flexible-server>.postgres.database.azure.com"
$targetPort = "5432"
$targetDb = "petclinic"
$targetUser = "petclinic_app"
$dumpPath = ".\evidence\logs\petclinic_pg_dump.dump"
```

Capture source facts:

```powershell
psql "host=$sourceHost port=$sourcePort dbname=$sourceDb user=$sourceUser sslmode=require" -c "SELECT version();"
psql "host=$sourceHost port=$sourcePort dbname=$sourceDb user=$sourceUser sslmode=require" -c "SELECT pg_size_pretty(pg_database_size('petclinic'));"
```

Stop writes:

1. Announce maintenance window.
2. Disable application write access or stop the application.
3. Confirm no active application write sessions.

Create dump:

```powershell
pg_dump --format=custom --verbose --no-owner --no-privileges --host $sourceHost --port $sourcePort --username $sourceUser --dbname $sourceDb --file $dumpPath
```

Restore to target:

```powershell
pg_restore --verbose --clean --if-exists --no-owner --no-privileges --host $targetHost --port $targetPort --username $targetUser --dbname $targetDb $dumpPath
```

Run validation:

```powershell
python .\scripts\data_validate\validate_petclinic_data.py --output-dir .\evidence\logs
```

For live database validation, run `scripts/data_validate/validation_queries.sql` against both source and target and compare the output.

## Cutover Sequence

1. Freeze application writes on source.
2. Confirm source backup/PITR point.
3. Run final `pg_dump`.
4. Restore to Azure PostgreSQL Flexible Server.
5. Run row count, checksum, FK and sequence validation.
6. Store target connection values in Key Vault or platform secrets.
7. Repoint application settings:
   - `SPRING_PROFILES_ACTIVE=postgres`
   - `POSTGRES_URL=jdbc:postgresql://<server>.postgres.database.azure.com:5432/petclinic?sslmode=require`
   - `POSTGRES_USER=<managed-db-user>`
   - `POSTGRES_PASS=<secret reference>`
8. Restart or roll a new application revision.
9. Run application smoke tests:
   - `/actuator/health`
   - `/owners/find`
   - `/vets.html`
   - create/read validation for a controlled test owner if writes are enabled.
10. Enable writes on target.
11. Start hypercare monitoring.

## Rollback Path

Keep the source database available for at least 24 hours after cutover.

| Trigger | Rollback Decision |
|---|---|
| Error rate above 2% for 10 minutes | Pause writes and evaluate rollback |
| p95 DB latency above 500 ms for 15 minutes | Check Azure metrics, connection pool and query waits |
| Data drift detected by validation harness | Stop writes and restore source as system of record |
| Application cannot connect to target | Repoint secrets to source and restart prior revision |

Rollback steps:

1. Stop writes on the target.
2. Confirm the source still has the pre-cutover consistent state.
3. Repoint application secrets back to source connection strings.
4. Restart the previous application revision.
5. Run smoke tests.
6. Preserve failed target state for root-cause analysis.

Reverse replication is not configured for the offline assessment path. For production near-zero-downtime migrations, configure logical replication or DMS online mode before cutover if rollback must preserve writes made after target promotion.

## Operational Configuration

| Area | Target Configuration |
|---|---|
| Private connectivity | Prefer private endpoint or delegated VNet access from the application environment |
| Firewall | Deny public access unless temporary migration runner access is explicitly required |
| TLS | Require TLS and use JDBC `sslmode=require` |
| Authentication | Use Microsoft Entra authentication where supported; store local database credentials in Key Vault when password auth is required |
| Backup/PITR | Enable automated backups and set retention according to business RPO |
| HA | Use zone-redundant HA for production if region/SKU supports it |
| Geo-DR | Use geo-redundant backups or read replica strategy for higher DR requirements |
| Maintenance | Define maintenance window outside business peak |
| Monitoring | Alert on CPU, storage, connections, failed connections, deadlocks, replication lag and slow queries |
